# 向 Argo CD 提交程式碼貢獻

## 前言

Argo CD 專案在功能和社群規模方面都在不斷增長。越來越多的組織採用它，並信任 Argo CD 來處理其關鍵的生產工作負載。因此，我們需要非常謹慎地對待任何影響 Argo CD 相容性、效能、可擴展性、穩定性和安全性的變更。因此，每一項新功能或較大的增強功能都必須在被接受到程式碼庫之前經過適當的設計和討論。

我們歡迎並鼓勵每個人參與 Argo CD 專案，但請理解，由於各種原因，我們無法接受社群的每一項貢獻。

如果您想為一個很棒的新功能或增強功能提交程式碼，我們懇請您在開始編寫程式碼或提交 PR 之前，先看一下下面概述的增強流程。這將確保您的想法與專案的策略和技術要求保持一致，並將大大有助於將您的程式碼合併到我們的程式碼庫中。

在為新功能（在某種程度上，也包括更複雜的錯誤修復）提交程式碼之前，請
先[提出一個增強提案或錯誤議題](https://github.com/argoproj/argo-cd/issues/new/choose)。

_請_在相應的增強功能被分類之前，不要在較大的功能或重構上花費太多時間。這可能會為每個人節省一些挫折和時間，因為增強提案可能會被拒絕，而程式碼將永遠不會被合併。然而，有時在提案的同時附上一些概念驗證 (PoC) 程式碼會很有幫助。

我們將盡最大努力快速對收到的增強提案進行分類，結果將是以下之一：

*   已接受
*   已拒絕
*   提案需要一份設計文件以供進一步討論

根據我們在特定時間收到的增強提案數量，可能需要一些時間才能輪到您的提案。

另外，請確保您已閱讀我們的
[工具鏈指南](toolchain-guide.md)
以了解我們的工具鏈和持續整合流程。它包含一些寶貴的資訊，可幫助您開始使用構成 Argo CD 的複雜程式碼庫。

## 快速入門

如果您想快速開始為 Argo CD 做出貢獻，請查看標有
[需要幫助](https://github.com/argoproj/argo-cd/issues?q=is%3Aopen+is%3Aissue+label%3A%22help+wanted%22)
或
[好的第一個議題](https://github.com/argoproj/argo-cd/issues?q=is%3Aopen+is%3Aissue+label%3A%22good+first+issue%22)
的議題。

這些是已經經過分類並被接受的議題。

如果該議題已經附加到下一個
[版本里程碑](https://github.com/argoproj/argo-cd/milestones)，
我們也決定將我們的一些審查時間用於為這些議題收到的 PR。

我們鼓勵我們的社群去處理那些被如此標記*且*附加到下一個版本里程碑的議題，並承諾它們將得到適當的審查，明確的意圖是讓收到的 PR 被合併。

## 分類流程

### 概覽

我們對增強提案的分類流程確保我們會審查所有收到的增強功能，以確定我們是否會接受實作它們的程式碼提交。

該流程如下：

*   在我們的 GitHub 議題追蹤器上提出的新增強提案會被移動到專案看板的 _Incoming_ 欄位。這些是等待分類的提案。
*   _Active_ 欄位存放目前正在分類或將要分類的議題。
*   _Accepted_ 欄位存放已經過分類並被認為可以實作的議題（例如，專案同意該功能將會很棒）
*   _Declined_ 欄位存放 triage 期間被拒絕的議題。該議題將會更新，說明提案被拒絕的原因。
*   _Needs discussion_ 欄位存放 triage 期間發現需要額外資訊，甚至需要設計文件的議題。

### 分類節奏

增強提案的分類是透明地進行的，可以透過議題評論離線進行，也可以在我們每週的貢獻者會議上線上進行。_每個人_都被邀請參與分類，該過程不僅限於維護者的參與。

通常，我們將以先進先出 (First-In-First-Out) 的順序對增強提案進行分類，這意味著最舊的提案將首先被分類。

我們的目標是每週至少分類 10 個提案。根據我們的可用時間，我們在任何給定的一周內可能會分類更多或更少的提案。

## 提案狀態

### 已接受的提案

當一個提案被認為是_已接受_時，這表示我們決定此增強功能對整個社群都很有價值，並且符合專案的整體策略藍圖。

該議題的實作可以開始，可以由提案的創建者或其他社群成員（包括專案的維護者）進行。

到目前為止，該議題應該已經被足夠完善，包含了在實作過程中需要考慮的任何顧慮和指導方針。

### 已拒絕的提案

我們不會輕易拒絕提案，並且會盡力給出適當的理由，說明我們為什麼認為該提案不符合專案的未來。拒絕提案的原因可能包括（但不限於）該變更對許多人來說是破壞性的，或者它不符合專案的策略方向。通常，在拒絕提案之前，會與增強功能的創建者進行討論。

一旦提案處於_已拒絕_狀態，我們就不太可能接受其實作的程式碼貢獻。

### 需要討論的提案

有時，我們無法從其 GitHub 議題中完全理解一個提案，需要更多關於原始意圖或實作細節的資訊。如果我們在分類過程中遇到這樣的議題，我們會將此議題移動到_需要討論_欄位，以表明我們希望議題的創建者提供更多關於他們想法的資訊。我們可能會要求您提供這些資訊，可以透過將資訊添加到議題本身，或加入我們的
[定期貢獻者會議](#regular-contributor-meeting)
與我們討論該提案。

此外，我們認為需要更正式設計文件的議題也將被移動到此欄位。

## 設計文件

對於某些增強提案（特別是那些將會大幅改變 Argo CD 行為、附帶一些警告或升級/降級路徑不清楚的提案），將需要一份更正式的設計文件，以便在更廣泛的社群中充分討論和理解該增強功能。此要求通常在分類期間確定。如果您提交了增強提案，我們可能會要求您提供這份更正式的書面文件，以及一些需要解決的顧慮或主題。

設計文件通常以 PR 的形式提交，並使用[此範本](https://github.com/argoproj/argo-cd/blob/master/docs/proposals/001-proposal-template.md)作為我們正在尋找的資訊類型的指南。討論將在審查過程中進行。當設計文件被合併時，我們認為它已獲批准，並且可以編寫和提交程式碼來實作此特定設計。

## 定期貢獻者會議

我們的社群定期舉行線上會議，討論圍繞 Argo CD 的議題、想法和增強功能。如果您想提出某些事情（包括您的增強提案）、參與我們的分類，或者只是想認識其他貢獻者，我們誠摯地邀請您參加這些線上會議。

我們目前的會議頻率是每週一次，時間是太平洋時間每週四上午 8:15（[點擊這裡查看您目前所在時區的時間][1]）。我們使用 Zoom 進行這些會議。

*   [議程文件（Google 文件，包含 Zoom 連結）](https://docs.google.com/document/d/1xkoFkVviB70YBzSEa4bDnu-rUZ1sIFtwKKG1Uw8XsY8)

如果您想討論某些事情，我們懇請您將您的項目放在
即將召開的會議的[議程](https://docs.google.com/document/d/1xkoFkVviB70YBzSEa4bDnu-rUZ1sIFtwKKG1Uw8XsY8)
上，以便我們可以安排時間進行討論。
